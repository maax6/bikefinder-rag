Bikefinder RAG · note d'ingénierie

Un agent RAG sans framework : la boucle est le produit

Bikefinder répond à « un roadster léger en A2, et qu'en disent les propriétaires ? » en interrogeant 14 000 fiches techniques et des dizaines de milliers d'avis de forums. Le cœur de l'agent tient en une boucle de ~40 lignes écrite à la main — pas de LangChain, pas de LlamaIndex. Cette page montre la différence, et ce que la transparence a rapporté concrètement.

Le processus, sans magie

Un « agent » n'est rien d'autre que ceci : un modèle qui reçoit des schémas d'outils, décide d'appeler un outil ou de répondre, et revoit passer le résultat de l'outil comme un message. Tout le reste est de l'emballage.

Bikefinder agent/loop.py — le vrai code
messages.append({"role": "user", "content": user_message})

for _ in range(MAX_TOOL_TURNS):          # jamais de boucle infinie
    response = client.messages.create(
        model=MODEL, max_tokens=2048,
        system=SYSTEM_PROMPT, tools=TOOLS,
        messages=messages,
    )
    messages.append({"role": "assistant",
                     "content": response.content})

    if response.stop_reason != "tool_use":  # réponse finale
        return final_text(response), messages

    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            results = execute_tool(conn, block.name, block.input)
            tool_results.append(as_tool_result(block.id, results))

    messages.append({"role": "user", "content": tool_results})

Même longueur, ou presque. La différence n'est pas le nombre de lignes — c'est qui possède l'état. Ici, l'historique messages est une liste Python qu'on peut lire, logger, tronquer et rejouer. Avec un framework, il vit dans un agent_scratchpad dont le format est un détail d'implémentation.

Ce que la transparence a payé — quatre cas réels du projet

Les modèles locaux dérivent du schéma d'outils

mistral-small envoie parfois models au lieu de model, ou oublie l'argument requis query. Comme execute_tool est à nous, l'erreur devient un tool_result lisible que le modèle corrige au tour suivant — et un query manquant est synthétisé depuis marque/modèle au lieu d'échouer.

faithfulness narratives : 0.0 → 0.83–1.0
L'éval lit les messages bruts

RAGAS a besoin des « contexts » (ce que le retrieval a vraiment renvoyé). Ici, on les extrait de l'historique messages en 15 lignes — les tool_result y sont en clair. C'est en lisant ces messages qu'on a vu que « GSF 1200 Bandit » ne matchait pas « GSF 1200 N Bandit ».

2 bugs de retrieval trouvés par l'éval
Un second backend en 30 lignes

Pas de clé API ? La même boucle tourne sur Ollama (Mistral local) : seule la forme des messages change. Le dev, les tests et l'éval RAGAS complète tournent gratuitement en local — le juge aussi (qwen3.6).

éval 100 % locale, 0 € d'API
Des garde-fous exacts, pas génériques

Plafond de 8 tours d'outils (un modèle qui boucle brûlerait la clé du visiteur), messages d'erreur métier pour clé rejetée ou serveur local éteint, max_tokens choisi en connaissance de cause. Chaque garde-fou est une ligne qu'on voit, pas une option de config à deviner.

démo publique BYOK sans stacktrace

Le process complet, de la page web à la réponse évaluée

01
Scraping poli
bikez.com, délai configurable, backoff auto sur 429/5xx, reprise au thread près. ~14 000 fiches + forums complets.
02
Modèle de données
Postgres + pgvector. Les avis appartiennent aux familles de modèles (le forum bikez est partagé entre millésimes), dédupliqués par hash.
03
Embeddings locaux
BGE-M3 (multilingue) : une question en français retrouve des avis en anglais. Aucune clé d'API d'embeddings.
04
Deux outils typés
filter_specs (SQL paramétré, tri whitelisté) et search_reviews (cosinus pgvector + filtres mot-à-mot).
05
La boucle
Celle du schéma ci-dessus. Claude en prod, Ollama en dev — même code, même historique inspectable.
06
Éval à deux étages
Retrieval seul sans LLM (lifts 21-82×), puis RAGAS sur l'agent complet : faithfulness 0.78 / relevancy 0.87 en full local.

Ce qu'on perd, honnêtement

Écrire la boucle soi-même n'est pas toujours le bon choix. Un framework se justifie quand :

  • le streaming, les retries, le tracing doivent exister dès le premier jour — ici ils sont ajoutés au besoin, à la main ;
  • on branche beaucoup de connecteurs (30 sources de données, 10 vector stores) — l'écosystème LangChain les a déjà écrits ;
  • l'équipe vit déjà dans LangSmith/LangGraph et son outillage d'observabilité.

Pour un système à 2 outils et 1 base, dont l'objectif est de comprendre et de démontrer chaque maillon, l'abstraction coûtait plus qu'elle ne rapportait. Ironie vérifiable : la seule dépendance LangChain du projet est arrivée via RAGAS — et c'est elle qui a cassé l'éval (conflit de versions, épinglé dans pyproject.toml).
